# Observabilidade em Next.js: Logging Consistente com LogLayer

## Por que precisamos falar de logging no Next.js?

Se voce ja tentou adicionar logging estruturado em um projeto Next.js moderno (App Router, server/client components, edge middleware), provavelmente percebeu que o framework nao oferece muito suporte nativo. Cada ambiente - client, server, edge - tem seu proprio runtime e limitacoes, e integrar uma solucao unificada de observabilidade e menos trivial do que parece.

Este artigo e uma tentativa direta de consolidar os aprendizados que tivemos ao implementar logging com **LogLayer**, **Pino** e integracao com **Datadog**, resolvendo os desafios reais de ambientes hibridos e serverless.

---

### Por que LogLayer?

A escolha do LogLayer se fundamenta em tres pilares: abstracao unificada, extensibilidade e reducao de manutencao.

- Suporte nativo para multiplos transports (Pino, Datadog, Console)
- API unica e segura para server/client
- Detecta automaticamente o runtime
- Serializacao consistente de erros e stack traces
- Sistema de plugins e suporte a OpenTelemetry

Resultado: reducao de centenas de linhas de codigo customizado, menor custo de manutencao e integracao pronta para ambientes observaveis.

Beneficios mensuraveis:

- Troca de providers sem refatoracao
- Conformidade com OpenTelemetry e padroes de mercado
- Implementacao em horas (nao semanas)
- Observabilidade real desde o dia zero

---

### Limitacoes do Next.js

- Nao ha abstracao oficial de logging
- `console.*` nao e estruturado, nem serializa erros corretamente
- Nao ha correlacao com APM (Datadog) ou RUM
- Codigo server/client compartilham modulos - imports condicionais quebram build
- Transports diferentes por runtime precisam de orquestracao
- Importacoes de dependencias Node.js no client causam erros de hydration

---

### Abordagem adotada

- Substituicao completa de `console.*` por LogLayer
- Uso de `getLogger()` com runtime detection
- Transports controlados por `enabled: boolean`
- Nenhum import condicional baseado em `process.browser` ou `typeof window`
- Garantia de que codigo Node nunca vai para o client
- Logging orientado a metadata (`withMetadata()`)

---

### Implementacao pratica com LogLayer

```ts
// src/shared/libs/logger/index.ts
import { LogLayer } from 'loglayer';
import { PinoTransport } from '@loglayer/transport-pino';
import { DataDogTransport } from '@loglayer/transport-datadog';
import { DataDogBrowserLogsTransport } from '@loglayer/transport-datadog-browser-logs';
import { getSimplePrettyTerminal } from '@loglayer/transport-simple-pretty-terminal';
import { isClient, isServer } from '@/shared/utils/runtime';
import { serializeError } from 'serialize-error';

const log = new LogLayer({
  errorFieldName: 'error',
  errorSerializer: serializeError,
  transport: [
    getSimplePrettyTerminal({
      enabled: process.env.NODE_ENV === 'development',
      runtime: isServer ? 'node' : 'browser',
      viewMode: isServer ? 'inline' : 'message-only',
      includeDataInBrowserConsole: isClient,
    }),
    new PinoTransport({
      enabled: isServer && process.env.NODE_ENV === 'production',
      logger: pino({ level: 'trace' }),
    }),
    new DataDogBrowserLogsTransport({
      enabled: isClient,
      logger: datadogLogs.logger,
    }),
    new DataDogTransport({
      enabled:
        isServer &&
        process.env.NODE_ENV === 'production' &&
        !!process.env.DD_API_KEY,
      options: {
        ddClientConf: {
          authMethods: { apiKeyAuth: process.env.DD_API_KEY || '' },
        },
        ddServerConf: {
          site: process.env.DD_SITE || 'datadoghq.com',
        },
      },
    }),
  ],
  plugins: [
    {
      onBeforeMessageOut(params) {
        const tag = isServer ? 'Server' : 'Client';
        if (params.messages?.length && typeof params.messages[0] === 'string') {
          params.messages[0] = `[${tag}] ${params.messages[0]}`;
        }
        return params.messages;
      },
    },
  ],
});

export function getLogger() {
  return log.withContext({ isServer });
}
```

### Substituindo `console.*`

```ts
// src/shared/libs/logger/register.ts
if (process.env.NEXT_RUNTIME === 'nodejs') {
  console.error = createConsoleMethod(log, 'error');
  console.log = createConsoleMethod(log, 'log');
  console.info = createConsoleMethod(log, 'info');
  console.warn = createConsoleMethod(log, 'warn');
  console.debug = createConsoleMethod(log, 'debug');
}
```

### Integracao com Datadog

- `dd-trace` com `logInjection: true` habilita correlacao entre APM e logs
- Os campos `trace_id`, `span_id` e `session_id` sao propagados para Datadog Logs
- Para o browser, `@datadog/browser-logs` e RUM fazem o link via `session_id`

```ts
// Datadog APM init
import ddTrace from 'dd-trace';

const tracer = ddTrace.init({
  env: process.env.DD_ENV,
  service: process.env.DD_SERVICE,
  version: process.env.DD_VERSION,
  runtimeMetrics: true,
  logInjection: true,
});
```

---

### Conclusao

O Next.js nao oferece uma solucao nativa de logging estruturado para ambientes hibridos. Adotar LogLayer resolve esse gap de forma segura, extensivel e com padroes da industria. Com isso, transformamos `console.*` em uma base real de observabilidade, sem riscos de hydration, e com total integracao a plataformas como Datadog.
